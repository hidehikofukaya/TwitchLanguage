# TwitchLaunguage 設計仕様書

> 最終更新: 2026-03-13 (先読みキャッシュ・低残量通知を追加)

---

## 1. プロジェクト概要

Twitchのライブ配信チャットを素材として語学学習ができるChrome拡張機能。
流れるコメントから有用なフレーズ（単語・熟語・慣用句）を自動抽出し、母国語による解説・使用例をUIに表示する。

### コアバリュー
- 「翻訳して終わり」ではなく、スラング・ミーム・口語表現の**解説**まで踏み込む
- Twitchチャット特有の文化的文脈を含めた学習体験
- Language Reactor / Migaku が対応していない**ライブ配信**領域を開拓

---

## 2. ターゲットユーザー

- 外国語の配信を見たいが言語の壁がある視聴者
- ゲーム・Vtuber・雑談配信を通じて自然な口語表現を学びたい人
- 日本語話者をファーストターゲットとする

---

## 3. 収益モデル

### 基本方針
- **完全無料で開始可能**（会員登録不要でインストールのみ）
- 解説表示はコイン消費制（前払い）
- サブスクリプションは当面実装しない

### コインシステム

| アクション | コイン増減 |
|---|---|
| 会員登録完了 | **+10枚** |
| コインパック購入 | +購入枚数 |
| 解説表示（成功時） | **−1枚** |
| 解説表示（失敗時） | **±0枚**（消費しない） |

**コイン消費のタイミング：解説の表示処理が成功した時点で消費する。APIエラー・ネットワークエラー時は消費しない。**

### コインパック（Stripe Payment Links）

| パック名 | 価格 | 枚数 | 単価 |
|---|---|---|---|
| お試し | $1.99 | 30枚 | $0.066/枚 |
| スタンダード | $4.99 | 100枚 | $0.050/枚 |
| ヘビー | $9.99 | 250枚 | $0.040/枚 |

### 初期コイン枚数について
- 10枚に固定（20〜30枚にするとメールアドレスを変えた永久無料利用が可能になるため）

## 4. ユーザー体験フロー

```
1. 拡張機能インストール
       ↓
2. 拡張機能を有効化 → 設定画面が開く
   ・母国語を選択（例：日本語）
   ・翻訳対象言語を選択（例：英語）
   ※両方を同じ言語にすることも可能
       ↓
3. 会員登録（Google OAuth） → コイン10枚付与
       ↓
4. Twitchで配信を開く
       ↓
5. 拡張機能がチャットコメントの監視を自動開始
   （バックグラウンドでフレーズ収集＆キャッシュ生成が並行動作）
       ↓
6. 設定した間隔で解説が自動表示される → コイン1消費
   表示例：「W rizz｜すごいモテ力。"rizz"はカリスマ・魅力の意。
            最近のスラングで"W"は"winning"＝最高の。使用例: "bro has W rizz fr"」
       ↓
7. 「次のフレーズへ」ボタンで即時次の解説へ → コイン1消費
       ↓
8. コインが0になったら購入モーダルを表示
```

## 5. フレーズの定義

- **対象：** 単語・熟語・慣用句・スラング・Twitch固有ミーム
- **非対象：** 文章全体の翻訳（センテンス翻訳はスコープ外）
- **除外フィルタ：**
  - 3文字以下の極短語（"ok", "no" など）
  - 数字のみ・URLを含む文字列
  - Twitchエモート名単体（Kappa, PogChamp 等）→ 別途固定解説テーブルで管理

---

## 6. フレーズ抽出パイプライン

```
[Twitchチャット]
    ↓ MutationObserver / Twitch EventSub WebSocket
[コメントバッファ]
  直近50〜100件をスライディングウィンドウで保持
    ↓
[フレーズ候補抽出]
  ・コメント群から学習対象フレーズを切り出す
  ・除外フィルタ適用
    ↓
[スコアリング]
  ・出現頻度スコア（直近5分間の登場回数）
  ・スラングスコア（Urban Dictionary APIヒット率）
  ・新規性スコア（キャッシュに未登録 = 加点）
  ・選定回数ペナルティ（過去の選定頻度が高い = 減点 ← 後述）
    ↓
[バッチAPIキュー]
  上位フレーズをキューに積む（次のLLM呼び出しタイミングで処理）
```

## 7. LLM呼び出し戦略

### 基本方針：先読みキャッシュ生成

LLMの呼び出しは**表示タイミングとは完全に独立したバックグラウンド処理**として動作する。
表示側はキャッシュから取り出すだけなので、ユーザーは待ち時間ゼロで解説を受け取れる。

```
[バックグラウンドワーカー]              [表示タイマー]
       |                                      |
       | 2分ごと（またはキャッシュ低残量時）  | 設定間隔ごと
       ↓                                      ↓
  LLM呼び出し                         キャッシュキューから先頭を取り出す
  → 解説5〜10件生成                    → UIに表示 → コイン消費
  → キャッシュキューに追加
```

### 呼び出し頻度
- **通常：最大2分に1回**（API過剰呼び出し防止）
- **キャッシュ低残量時：早期トリガー**（後述）
- 表示頻度とLLM呼び出し頻度は独立している

### タイムライン例（表示間隔60秒・LLM2分周期の場合）

```
0s   → LLM呼び出し開始（バックグラウンド）→ 解説8件をキャッシュキューに積む
0s   → キャッシュキュー先頭を表示（コイン消費）    残り7件
60s  → キャッシュキュー先頭を表示（コイン消費）    残り6件
120s → LLM呼び出し開始（バックグラウンド）→ さらに8件追加    残り13件
120s → キャッシュキュー先頭を表示（コイン消費）
...
```

表示間隔が5分の場合は、5回の表示のうちLLM呼び出しは1回、キャッシュ利用は4回。

### キャッシュ低残量時の早期LLMトリガー

```
// キャッシュキューの残り件数を監視
if (cacheQueue.length <= LOW_CACHE_THRESHOLD) {
  // 通常の2分周期を待たずに即座にLLMを呼び出す
  triggerLLMEarly()
}
```

- `LOW_CACHE_THRESHOLD` の目安：**表示間隔 × 3 ÷ 60** 件
  - 例：表示間隔30秒なら残り1.5件 ≒ 2件を下回ったら早期トリガー
  - 例：表示間隔60秒なら残り3件を下回ったら早期トリガー
  - 考え方：「あと3回分の表示が切れる前にLLMを呼んでおく」

### UIへの状態通知

キャッシュの状態をオーバーレイUI上に小さいステータスとして常時表示する。

| 状態 | UIアイコン・表示 | 色 |
|---|---|---|
| キャッシュ十分（4件以上） | ● 準備完了 | グレー（目立たない） |
| LLM呼び出し中 | ⟳ フレーズを取得中... | 青・点滅 |
| キャッシュ低残量（2件以下） | ◐ フレーズを補充中... | 黄 |
| LLMエラー発生 | △ 取得に失敗しました | 赤・一時表示 |

- 通常時（キャッシュ十分）はステータスをほぼ非表示または極小表示にし、ユーザーの視聴を妨げない
- LLM呼び出し中・低残量時のみ視認できるサイズで表示する

### バッチプロンプト設計

- **送信するコンテキスト：** 直近50〜100件のコメントテキスト
- **1回で抽出するフレーズ数：** 精度を優先して最大5〜10フレーズ（精度が落ちるようなら5以下に絞る）
- **言語戦略：** プロンプトは英語・思考も英語・出力のみ母国語（トークン節約）

```
プロンプト例（英語）：
"You are a language learning assistant for Twitch viewers.
From the following Twitch chat comments, extract up to 5 phrases
(words, idioms, slang, or set expressions) that are worth learning.
For each phrase, provide in [NATIVE_LANGUAGE]:
  1. Translation/meaning
  2. Nuance and usage context
  3. Typical usage in Twitch chat (example sentence)
  4. Similar expressions (if any)

Respond ONLY in JSON format. Think in English, output values in [NATIVE_LANGUAGE].

Chat comments:
[50〜100件のコメント]"
```

**出力形式（JSON）：**
```json
[
  {
    "phrase": "W rizz",
    "translation": "すごいモテ力・カリスマ性",
    "nuance": "「W」は最高・勝ちを意味し、「rizz」は異性を惹きつける魅力のスラング",
    "example": "bro has W rizz fr → あいつ本当にモテすぎる",
    "similar": ["smooth talker", "big W"]
  }
]
```

## 8. キャッシュ設計

### 2層キャッシュ

| 層 | 場所 | 容量 | TTL | 用途 |
|---|---|---|---|---|
| **Layer 1** | chrome.storage.local | **10MB上限** | 30日 | 高速ローカル参照 |
| **Layer 2** | Supabase（サーバー） | 無制限 | 30日 | ユーザー間共有キャッシュ |

### キャッシュキー
- フレーズ文字列を正規化（lowercase・trim）したもの
- 言語ペア（母国語コード + 対象言語コード）を含める
- 例：`en-ja::W rizz`

### キャッシュ容量管理（Layer 1）

- **容量ベース管理**（エントリ数ではなくバイト数で制御）
- 10MBを超えたとき、**選定回数が多いフレーズ**から優先削除
  - 理由：よく表示されたフレーズ＝学習価値を使い切った可能性が高い
  - LRU（最近使われていないものを削除）は語学学習と相性が悪いため採用しない

```
// 削除優先スコア（高いほど削除対象）
eviction_score = selection_count × (1 / days_since_added)
```

## 9. フレーズ選定ロジック（表示時）

### 基本方針
- **加重ランダム選定**を採用
- 選定回数が多いフレーズほど選ばれにくくなる

### 選定確率の計算

```
base_score = frequency_in_chat × slang_score × novelty_bonus

// 選定回数ペナルティ（対数減衰）
penalty = 1 / (1 + log(selection_count + 1))

final_weight = base_score × penalty

// 全候補の final_weight を正規化して確率分布とする
```

### 同一フレーズの連続抑制
- 同じフレーズは**最低5回、別のフレーズを挟む**まで選定しない
- 他にフレーズ候補が存在しない場合に限り、重複選定を許可

### 選定回数の記録
- フレーズごとの選定回数を `chrome.storage.local` にキャッシュ
- サーバー側にも同期（端末をまたいだ体験一貫性のため）

## 10. 表示UI仕様

- Twitch画面にオーバーレイ表示（ドラッグ移動可能）
- 表示内容：フレーズ・訳・ニュアンス・使用例
- 表示間隔：ユーザー設定で調節可能（デフォルト60秒）
- 「次のフレーズへ」ボタン：コイン1消費で即時次の解説へ
- コイン残量を常時表示
- キャッシュ状態ステータス（小）を常時表示（Section 7参照）

---

## 11. オフライン・エラー時の挙動

| 状態 | 挙動 |
|---|---|
| オフライン | 機能無効化（コメント取得不可のため使用不可） |
| APIエラー（LLM側） | コイン消費しない。キャッシュからフォールバック表示を試みる |
| キャッシュ枯渇 + APIエラー | 「フレーズを取得できませんでした」を表示、コイン消費しない |
| コイン0 | 購入モーダルを表示 |

## 12. システムアーキテクチャ

```
Chrome Extension (Manifest V3)
├── content_script.js     Twitch DOM監視・フレーズ抽出・UI表示
├── service_worker.js     タイマー制御・API呼び出し・コイン管理
├── popup.html            設定（表示頻度・言語）・コイン残量・購入ボタン
└── chrome.storage.local  フレーズキャッシュ・選定回数・JWTキャッシュ

         ↕ HTTPS

Cloudflare Workers（APIゲートウェイ・OpenAI APIキーを隠蔽）
├── POST /phrases/batch   バッチ解説生成（GPT-4o-mini呼び出し）
├── GET  /coins/balance   コイン残量照会
├── POST /coins/consume   コイン消費（正はSupabase DB）
├── POST /auth/register   会員登録 → 初回コイン10枚付与
└── POST /webhook/stripe  購入完了 → Supabase にコイン加算

         ↕

Supabase（DB・認証）
├── users                 ユーザー情報（Google OAuthで認証）
├── coin_balances         コイン残量（改ざん防止の正データ）
├── phrase_cache          ユーザー間共有キャッシュ（言語ペア別）
├── selection_history     フレーズ選定回数記録
└── coin_transactions     購入・消費履歴（監査ログ）

         ↕ Webhook

Stripe
└── Payment Links         コインパック購入
```

## 13. セキュリティ設計

| 項目 | 方針 |
|---|---|
| OpenAI APIキー | Cloudflare Workers の環境変数に格納。拡張機能JSには含めない |
| コイン残量の正データ | Supabase DB。ローカルはUI表示用キャッシュのみ |
| JWT保存 | chrome.storage.local に有効期限付きで保存（syncは使わない） |
| コイン改ざん対策 | サーバー側でコイン減算後にOpenAI APIを呼び出す（順序が重要） |
| Stripe Webhook冪等性 | stripe_event_id をDBに記録し重複処理を防ぐ |

## 15. 実装フェーズ

### Phase 1（MVP）
- MutationObserver によるTwitchコメント取得
- GPT-4o-miniバッチ呼び出し＋解説表示
- コイン制（Supabase + Cloudflare Workers）
- Stripe コインパック購入
- 基本UIオーバーレイ

### Phase 2
- 選定回数ベースの加重ランダム選定
- フレーズキャッシュ最適化（10MB管理・選定回数ベース削除）
- 表示間隔の設定機能
- 「次のフレーズへ」ボタン

### Phase 3（応用機能・検討中）
- コメントのホバー即時翻訳
- 配信者音声の字幕連携（技術・権利面の課題あり、要調査）
